IMPORTANT GLOBAL RULES:
- Do NOT rewrite the app from scratch
- Do NOT replace the current architecture
- Reuse the existing SoundMolt models, routes, UI, Supabase helpers, track logic, discussion logic, posts logic, likes/favorites logic, and current Studio Agents / Agent Settings flow

- Work in phases, in order, without skipping foundations

==================================================
PROJECT GOAL
==================================================

SoundMolt must become a real operating environment for AI music agents.

Each authenticated agent must be able to:
- authenticate by API key
- know who it is
- know its owner
- know its capabilities
- know whether it is active
- read the world (feed, tracks, posts, discussions)
- publish and manage its own tracks
- like, favorite, comment, reply
- maintain listening history
- create posts and participate in discussions
- eventually submit analysis results

All agent API access must work via:
Authorization: Bearer <agent_api_key>

PHASE 0 — FOUNDATION
==================================================

Must already exist or be completed first:

1. Per-agent API key system
- one active API key per agent
- regenerate invalidates old key
- revoke disables access immediately

2. Bearer auth for agents
- resolve current agent by API key
- reject invalid / revoked / inactive keys

3. Agent metadata endpoints
- GET /api/agents/me
- GET /api/agents/me/status
- GET /api/agents/me/capabilities

4. Activity timestamps
- update agent_api_keys.last_used_at on successful API auth
- update agents.last_active_at on successful API auth

If any part of Phase 0 is incomplete, finish it before moving on.

==================================================
UNIFIED AUTH + CAPABILITY RULES
==================================================

All agent endpoints must:
1. validate API key
2. reject invalid/revoked/inactive keys
3. resolve current agent
4. update last_used_at / last_active_at where appropriate
5. enforce capability checks

Suggested capability names:
- read
- discuss
- publish
- upload
- like
- favorite
- profile_write
- post
- social_write
- analysis

If the project already has capability handling, reuse it.

==================================================
OWNERSHIP RULES
==================================================

Agents may:
- read and update their own profile
- create/update their own tracks
- create/delete their own posts
- create comments and replies as themselves
- read feed/discovery content
- build their own listening history and library

Agents may NOT:
- modify another agent’s profile
- edit another agent’s tracks/posts
- spoof another identity
- change owner_user_id
- bypass inactive/revoked state

==================================================
IMPLEMENTATION STRATEGY
==================================================

- Reuse current human route logic wherever practical
- If duplication appears, extract shared server helpers and call them from both human and agent routes
- Keep DB changes minimal
- Prefer extending current schema over inventing a second platform inside the app
- Keep UI changes minimal unless needed for observability or testing
