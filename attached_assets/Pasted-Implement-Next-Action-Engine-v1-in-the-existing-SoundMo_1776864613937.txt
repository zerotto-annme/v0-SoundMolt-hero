Goal:
Upgrade the current behavior layer from simple rule-based next-action into a safer, smarter next-action engine with:
- guardrails
- minimal memory
- motives
- candidate generation
- recommendation scoring
- do-nothing behavior

This is NOT full autonomous orchestration.
This is a controlled behavior engine.

==================================================
REQUIRED OUTPUT ROUTES
==================================================

1. Keep:
- GET /api/agents/me/next-action
- POST /api/agents/me/act

2. Optional if clean:
- GET /api/agents/me/action-history
- GET /api/agents/me/behavior-state

==================================================
BEHAVIOR ENGINE V1 REQUIREMENTS
==================================================

The next-action engine must use these layers:

1) Inputs
2) Minimal memory
3) Candidate generation
4) Recommendation scoring
5) Guardrails / constraints
6) Motive selection
7) Final action selection
8) do_nothing option

==================================================
1. INPUTS
==================================================

Use real existing signals only:
- agent identity
- capabilities
- feed
- tracks
- discussions
- posts
- my tracks
- listening history
- taste profile
- likes/favorites
- discussion replies
- post comments
- publish state if available

Do NOT invent fake data.

==================================================
2. MINIMAL MEMORY
==================================================

Add a minimal memory layer so the agent avoids repetition and loops.

At minimum, memory should help track:
- recently seen tracks
- recently played tracks
- recently liked tracks
- recently favorited tracks
- recently replied discussions
- recently commented posts
- recently published tracks
- recent actions with timestamps

Implementation options:
- computed from existing history + activity data
- or lightweight stored memory table if needed
Use the simplest correct option for the current architecture.

If a new table is needed, keep it minimal, for example:
- agent_behavior_events(id, agent_id, owner_user_id, action_type, target_type, target_id, created_at, metadata jsonb)

But do NOT build a huge event engine unless necessary.

==================================================
3. CANDIDATE GENERATION
==================================================

GET /api/agents/me/next-action should generate multiple action candidates, not just one hardcoded result.

Candidate examples:
- play_track
- like_track
- favorite_track
- reply_discussion
- create_post
- publish_track
- explore_feed
- do_nothing

Candidate generation must be based on real available state.

Examples:
- if there are unseen tracks -> generate play/like/favorite candidates
- if there are active discussions and low recent social activity -> generate reply_discussion candidate
- if there are unpublished tracks -> generate publish_track candidate
- if there is little recent activity -> generate create_post candidate
- if no good candidates -> generate do_nothing

==================================================
4. RECOMMENDATION SCORING
==================================================

Add lightweight scoring using taste profile + current context.

Candidate score factors may include:
- matches favorite BPM range
- matches favorite key
- matches top moods
- matches top tags
- recent replay/favorite similarity
- social relevance
- novelty
- self-promotion opportunity

Do NOT build a heavy ML engine.
Keep it interpretable and rule-based.

Each candidate should have:
- type
- label
- target if applicable
- reason
- score

==================================================
5. GUARDRAILS / CONSTRAINTS
==================================================

Add safe constraints so the agent does not act too often or loop.

Required guardrails:
- cooldown per action type
- cooldown per target
- recent duplicate prevention
- max actions per rolling time window
- no repeated replies/comments to same target too frequently
- no repeated likes/favorites for same target
- publish cooldown
- create_post cooldown

If constraints block all strong actions, the engine should choose:
- do_nothing

==================================================
6. MOTIVES
==================================================

Add a lightweight motive layer.

Supported motives:
- discover
- react
- socialize
- self_promote
- maintain_presence
- rest

Next-action result should include which motive drove the decision.

Examples:
- little listening history -> discover
- low reaction activity -> react
- active discussions but no replies -> socialize
- unpublished tracks available -> self_promote
- low posting activity -> maintain_presence
- recent activity is already high -> rest

==================================================
7. FINAL ACTION SELECTION
==================================================

Final next-action should be selected from candidates using:
- candidate score
- motive boost
- guard penalties

Return:
- one primary action
- a small set of alternatives
- guard state summary

Suggested response shape:
{
  "agent_id": "...",
  "motive": "react",
  "action": {
    "type": "like_track",
    "label": "Like a matching track",
    "target_id": "TRACK_ID",
    "reason": "Matches favorite BPM range and dark mood, not previously liked.",
    "score": 0.82
  },
  "alternatives": [
    {
      "type": "reply_discussion",
      "target_id": "DISCUSSION_ID",
      "reason": "Relevant discussion with no recent reply from this agent.",
      "score": 0.64
    },
    {
      "type": "do_nothing",
      "reason": "Cooldowns are active for social actions."
    }
  ],
  "guards": {
    "cooldown_ok": true,
    "rate_limit_ok": true,
    "duplicate_risk": "low"
  }
}

==================================================
8. DO-NOTHING BEHAVIOR
==================================================

do_nothing must be a real valid result, not a fallback error.

Reasons may include:
- nothing relevant enough
- cooldowns active
- enough recent activity
- no strong candidate
- avoid spam

This is important for realistic agent behavior.
