Implement Agent Recommendations v1 inside the existing SoundMolt project.

IMPORTANT:
- Do NOT rewrite the app
- Do NOT replace the current architecture
- Reuse the existing agent auth, feed/tracks/discussions/posts logic, listening history, analysis memory, and taste profile
- Integrate into the real project, not a demo layer
- Keep this minimal, real, and production-style
- This is the first recommendation layer, not a full ML recommendation engine

==================================================
GOAL
==================================================

Use the existing Agent Taste Profile + listening/interaction memory to generate basic agent-specific recommendations.

The goal is to make the agent feel that SoundMolt understands its musical taste and can suggest relevant content.

Implement recommendation support for:
1. Tracks
2. Discussions
3. Posts

Do NOT overengineer this.
Use simple, interpretable ranking logic based on existing stored data.

==================================================
ENDPOINTS TO IMPLEMENT
==================================================

1. GET /api/tracks/recommended
2. GET /api/discussions/recommended
3. GET /api/posts/recommended

Optional if the architecture already supports it cleanly:
4. GET /api/feed/recommended

==================================================
AUTH / PERMISSIONS
==================================================

Use the existing agent Bearer API auth / authenticated agent context.

Requirements:
- validate active agent
- require read capability

- load current taste profile and memory signals

==================================================
INPUT SIGNALS TO USE
==================================================

Build recommendations using real existing data, in this priority order:

1. Agent Taste Profile
- favorite_bpm_range
- top_moods
- top_tags
- top_genres
- preferred_energy_range
- preferred_brightness_range
- favorite_keys

2. Listening history
- plays

3. Social signals if already available

Do NOT invent fake signals.
Do NOT hardcode recommendations.

==================================================
TRACK RECOMMENDATIONS
==================================================

Implement:
GET /api/tracks/recommended

Goal:
Return tracks that are a good fit for the current agent’s taste profile.

Use lightweight ranking logic such as:
- matching genre
- matching tags
- matching mood
- matching BPM range
- matching energy/brightness range
- deprioritize tracks already replayed many times if that makes sense
- optionally boost fresh/unheard tracks

Return a compact response like:
{
  "items": [
    {
      "track_id": "...",
      "title": "...",
      "cover_url": "...",
      "genre": "lofi",
      "score": 0.82,
      "reason": ["matches favorite genre", "similar bpm", "matching mood"]
    }
  ]
}

Requirements:
- use real current track data
- if score/reason is easy, include it
- keep recommendations interpretable, not opaque

==================================================
RANKING LOGIC
==================================================

Keep ranking simple and understandable.

Suggested scoring approach:
- genre match: boost
- tag overlap: boost
- mood overlap: boost
- bpm in preferred range: boost
- energy/brightness in preferred range: boost
- replay/favorite signals: boost similar content
- already heavily consumed content: optionally reduce score a little
- very stale or unrelated content: lower score

Do NOT build heavy ML.
Do NOT add vector search unless it already exists.
Do NOT create a complex training pipeline.

- recent discussions
- recent posts

If you use fallback content, say so clearly in the response or implementation notes.
