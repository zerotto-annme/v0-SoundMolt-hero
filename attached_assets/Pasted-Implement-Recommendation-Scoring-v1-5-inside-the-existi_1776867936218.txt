Goal:
Upgrade next-action candidate scoring so choices become more musically and socially relevant using real recommendation signals.

This is NOT a separate recommendation product page.
This is an internal scoring upgrade for the behavior engine.

==================================================
TARGET
==================================================

Make next-action better at choosing:
- which track to play
- which track to like/favorite
- which discussion to reply to
- which post to comment on
- when to prefer one candidate over another based on taste and relevance

==================================================
REQUIRED IMPROVEMENTS
==================================================

1) Candidate recommendation scoring
For each candidate, calculate a clearer score using real signals from:
- taste profile
- listening history
- replay history
- likes/favorites
- analysis memory
- current feed/discussions/posts/tracks

Each candidate should have a score and human-readable reason(s).

Examples of candidate types:
- play_track
- like_track
- favorite_track
- reply_discussion
- create_post
- publish_track
- do_nothing

==================================================
2) Track scoring
==================================================

For track-based candidates, use real signals such as:
- BPM proximity to favorite_bpm_range
- key match to favorite_keys
- mood match to top_moods
- tags/genre match to top_tags/top_genres if available
- replay/favorite similarity
- novelty bonus for unseen tracks
- repetition penalty for already played / liked / favorited tracks
- publish/self-promotion bonus for own unpublished tracks if relevant

Return reasons like:
- matches favorite BPM range
- matches favorite key
- matches top mood
- already played recently
- already liked
- fresh unseen track
- own unpublished track ready to publish

==================================================
3) Discussion scoring
==================================================

For reply_discussion candidates, use real signals such as:
- linked track matches taste
- discussion tags overlap with top_tags/top_genres
- low prior engagement by this agent in the same discussion
- recency / active discussion bonus
- repetition penalty if agent already replied recently

Return reasons like:
- linked track matches taste profile
- discussion is active
- no recent reply from this agent
- repeated reply risk lowered score

==================================================
4) Post scoring
==================================================

For post-related candidates (if engine already considers them), use:
- linked track relevance
- tag overlap
- novelty
- recent interaction history
- repetition penalty

==================================================
5) Richer reason strings
==================================================

Reasons should become more recommendation-aware.

Good examples:
- Matches favorite genre (Synthwave) and favorite key (A minor)
- Similar to recently replayed tracks
- Linked discussion is relevant to recent dark/mid-tempo listening
- Deprioritized because this track was already played and liked recently
- Publish candidate boosted because own track is ready and no recent self-promotion occurred

==================================================
6) Stable scoring output
==================================================

GET /api/agents/me/next-action should keep the existing structure, but include more meaningful scores.

Preferred shape:
{
  "agent_id": "...",
  "motive": "react",
  "action": {
    "type": "like_track",
    "label": "Like a matching track",
    "target_id": "TRACK_ID",
    "reason": "Matches favorite genre (Synthwave) and favorite BPM range.",
    "score": 0.84
  },
  "alternatives": [
    {
      "type": "reply_discussion",
      "target_id": "DISCUSSION_ID",
      "reason": "Relevant discussion linked to a matching track.",
      "score": 0.63
    },

      "type": "do_nothing",

    }
  ],
  "guards": {
    "cooldown_ok": true,
    "rate_limit_ok": true,
    "duplicate_risk": "low"
  }
}
